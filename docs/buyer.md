# Buyer: согласованная спецификация v1

## Назначение

`buyer` — агент покупки, который получает задачу, открывает сайт магазина, проходит сценарий выбора и оформления товара, а затем передает данные для платежного шага в `middle`.

## Архитектурный контур

- `openclaw` передает high-level задачу в `buyer`.
- `buyer` напрямую управляет браузером через Playwright и выполняет сценарий покупки.
- `middle` выступает каналом общения с клиентом и интерфейсом для оплаты на iPhone.

## E2E-флоу v1

1. `buyer` получает задачу и ограничения.
2. Запускает браузерную сессию Playwright и открывает целевой магазин.
3. Находит товар, проверяет соответствие условиям, добавляет в корзину.
4. Проходит checkout до шага SberPay.
5. При необходимости уточнений отправляет запрос в `middle` и ждет ответ.
6. На шаге SberPay извлекает `orderId` и возвращает его в `middle`.
7. Завершает сценарий со статусом выполнения.

## Граница оплаты

- Ответственность `buyer` заканчивается на подготовке платежного шага и передаче `orderId`.
- Финальное подтверждение и проведение платежа происходят вне `buyer` в клиентском контуре.

## Концептуальные callback-события (без OpenAPI)

- `ask_user`: требуется уточнение от клиента (параметры товара, адрес, лимиты, замены и т.п.).
- `checkout_ready`: корзина и checkout подготовлены к финальному этапу.
- `payment_ready`: получен `orderId` и платежный шаг готов к отображению в `middle`.
- `handoff_requested`: агент запрашивает передачу управления человеку.
- `handoff_resumed`: ручной этап завершен, управление возвращено `buyer`.
- `scenario_finished`: итог сценария и артефакты выполнения.

## Контракт v1 `buyer` ↔ `middle` (без OpenAPI)

- Транспорт: HTTP callbacks с версионированным event envelope.
- Обязательные поля envelope: `event_id`, `session_id`, `event_type`, `occurred_at`, `idempotency_key`, `payload`.
- Семантика доставки: `at-least-once`; `middle` обязан дедуплицировать события по `event_id` и/или `idempotency_key`.
- Профиль доставки по умолчанию: `timeout=10s`, `3 retries`, exponential backoff с jitter; после исчерпания попыток событие помечается как failed и поднимается в сессионную ошибку.
- Отдельные auth-события в v1 не вводятся; auth-статусы и причины передаются в `scenario_finished` и `ask_user`.
- Канонические reason-коды auth:
  - `auth_ok`
  - `auth_failed_payload`
  - `auth_failed_redirect_loop`
  - `auth_failed_invalid_session`
  - `auth_refresh_requested`

## SberId авторизация v1

- Охват сайтов в v1: смешанный режим.
  - На поддерживаемых SberId-сайтах — авторизация через SberId.
  - На неподдерживаемых — guest-flow; если логин становится блокирующим, `buyer` переводит сценарий в handoff.
- Поддерживаемые SberId-сайты определяются через allowlist доменов.
- Владение auth-данными: `middle` является каноническим владельцем, `openclaw` работает как прокси при доставке задачи в `buyer`.
- Формат передачи auth-пакета: inline `storageState` в task payload.
- Жизненный цикл auth-пакета: только в памяти текущей сессии (`session-bound`), без постоянного хранения и reuse между сессиями.
- При битом/непарсящемся `storageState`:
  - `buyer` фиксирует `auth_failed_payload`,
  - отправляет `ask_user`,
  - запрашивает новый auth-пакет.
- Стратегия входа через SberId: сначала магазинные скрипты, затем эвристический fallback, затем handoff.
- Успех авторизации через SberId фиксируется при двух условиях:
  - произошел редирект обратно на домен магазина,
  - найден маркер авторизованного состояния на сайте.
- Ограничение redirect-loop при `id.sber.ru`: максимум 2 полных цикла; дальше фиксируется `auth_failed_redirect_loop`.
- Бюджет повторов auth: 1 повторный цикл с новым auth-пакетом, далее эскалация в handoff/остановка сценария.
- Проверка соответствия аккаунта пользователя после логина в v1 не выполняется.
- Фильтрация входящего `storageState` по доменам в v1 не выполняется.

## Скрипты Playwright

- Заготовленные сценарии оформляются как TypeScript-модули Playwright.
- Для известных доменов допускается `scripts-first` не только на авторизации, но и на покупке до платежного шага.
- Первый быстрый purchase-сценарий включен только для `litres.ru`: он должен вернуть `orderId` со страницы оплаты и не нажимать финальную кнопку платежа.
- Если быстрый purchase-скрипт не уверен в результате, `buyer` продолжает generic `codex exec` flow.
- Скрипты запускаются через единый контракт:
  - вход: контекст сессии, задача, ограничения;
  - выход: `status`, `artifacts`, `next_action`.
- После выполнения `buyer` может предлагать новый скрипт:
  - формирует кандидат-скрипт (черновик),
  - прикладывает краткую рекомендацию по применению,
  - не включает автоприменение по умолчанию.

## Handoff (VNC/noVNC)

- Включается по гибридной политике:
  - автоматически при блокерах (CAPTCHA, критические сбои, нестандартные UI-потоки),
  - вручную через `middle`/оператора.
- В MVP браузер для наблюдения разворачивается отдельным sidecar-контейнером (`Chromium + noVNC`), а `buyer` подключается к нему по CDP.
- Во время handoff логируются клики пользователя: координаты и селекторы.
- После ручного прохождения сложного шага `buyer` продолжает работу в той же браузерной сессии.
- CAPTCHA решается только человеком в handoff-режиме.

## Логирование

- Зафиксированная текущая политика: логи без ограничений во всех средах.
- Для SberId auth в v1 осознанно допускается логирование полного `storageState` и значений cookies для отладки.
- Trace шага включает агрегаты browser actions: суммарное время команд, idle между командами, количество/объем HTML и breakdown по командам.

## Архитектурные референсы и анти-цели

- Статус решения: принято, дата фиксации — 2026-04-23.
- `bitgn` используется как референс инженерных идей, но не как шаблон каркаса `buyer`.
- Что берем из `bitgn`: структурированные outcome-результаты, ретраи/таймауты, наблюдаемость, дисциплину системных инструкций и контекста.
- Что не переносим из `bitgn`: `vault_*`-модель работы с данными, файлово-ориентированный workflow, прямой перенос каркаса `codex exec` 1:1.
- Правило для дальнейшей подготовки требований: каждый новый requirement по `buyer` проверяется на соответствие этой договоренности; требования, которые скрыто тянут архитектуру в сторону полного порта `bitgn`, отклоняются или переформулируются под browser-buying домен.
- Базовый набор архитектурных решений для новых требований фиксируется в `docs/architecture-decisions.md`.
