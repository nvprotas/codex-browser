# AGENTS.md

## Статус документа

Этот файл является живым документом. Codex может изменять его по своему усмотрению для поддержания актуальности процессов, интерфейсов и правил работы агентной системы.

Любое изменение `AGENTS.md` должно сопровождаться записью в разделе `Журнал изменений` с датой, краткой причиной и перечислением ключевых правок.

## Локальные правила проекта

- Использовать русский язык для документации и комментариев.
- Для GitHub-операций использовать только `gh` и обычные git-команды.
- Не использовать GitHub connector / `codex_apps` для любых GitHub-операций.
- Если изменяется `.env.example`, необходимо синхронно обновлять `.env`.
- Перед изменением кода при необходимости читать `docs/repository-map.md`, чтобы понимать границы компонентов, входы, выходы и ошибки.
- При любом изменении кода обновлять `docs/repository-map.md` и поддерживать карту репозитория в актуальном состоянии.
- При локальном завершении feature-ветки вливать изменения в `master` через один итоговый squash-коммит и `git merge --ff-only`.
- При изменении roadmap, требований или фактического статуса работ поддерживать актуальное состояние задач в Linear: создавать недостающие issue, обновлять описания, статусы, зависимости, оценки и ссылки в документации.

## Эффективный запуск тестов

- Для Python-тестов `buyer` запускать pytest через `uv run` без префикса `PYTHONPATH=.`:
  - `uv run --with-requirements buyer/requirements.txt --with pytest pytest buyer/tests/test_cdp_recovery.py buyer/tests/test_observability_and_cdp_tool.py`
- Корневой `pytest.ini` уже добавляет `pythonpath = .`, поэтому отдельная переменная окружения не нужна.
- При точечной проверке указывать конкретные файлы или тесты после `pytest`, чтобы не запускать весь набор без необходимости.

## Цель системы

Разрабатывать агентного `buyer`, который может заходить на сайт любого магазина, проходить путь покупки и доводить сценарий до шага оплаты через SberPay.

## Роли и ответственность

- `openclaw`: ставит high-level задачу (что купить, ограничения, приоритеты).
- `buyer`: оркестрирует сценарий покупки и напрямую управляет браузером через Playwright.
- `middle`: ведет диалог с клиентом (включая iPhone-приложение), принимает артефакты от `buyer` и отображает платежный шаг.

## Граница платежа (SberPay)

- `buyer` доходит до шага оплаты через SberPay и возвращает в `middle` `orderId`.
- Финальный платеж и подтверждение транзакции выполняются вне `buyer` (через клиентский контур).

## Режимы работы buyer

- Автономный сценарий покупки без ручного вмешательства.
- Скриптовый режим: запуск заготовленных Playwright-скриптов.
- `handoff` режим через VNC/noVNC для сложных и нестабильных шагов.

## Скрипты и развитие базы сценариев

- Заготовленные сценарии хранятся как TypeScript-модули Playwright.
- `buyer` должен уметь предлагать новые скрипты после прохождения сценариев: формировать кандидат (черновик) и рекомендацию к применению.
- Автоприменение новых скриптов не является обязательным поведением по умолчанию.

## Handoff через VNC/noVNC

- Триггер входа в handoff гибридный:
  - автоматически при блокерах (включая CAPTCHA и критические ошибки),
  - вручную по команде из `middle`/оператора.
- В handoff логируются действия пользователя: координаты кликов и селекторы.
- После ручного шага `buyer` продолжает выполнение в той же браузерной сессии.
- CAPTCHA обрабатывается только через handoff человеком.

## Политика логирования

- Текущая зафиксированная политика: логи без ограничений во всех средах.
